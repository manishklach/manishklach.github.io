Start With the Problem: Tokens Need Context

A language model predicts the next token by looking at the tokens before it. The hard part is deciding which earlier tokens matter.

In the sentence The trophy does not fit in the suitcase because it is too large, a model must decide what it refers to. The answer depends on relationships between words, not just the words in isolation. Attention is the mechanism that lets every token compare itself with other tokens and pull in useful information.

Queries, Keys, and Values

A Transformer turns each token into a vector. Inside an attention layer, that vector is projected into three related vectors:

  • Query: what this token is looking for.
  • Key: what this token offers as searchable metadata.
  • Value: the information this token contributes if selected.
Attention(Q, K, V) = softmax(QK^T / sqrt(d)) V

The query is compared against keys. Those scores become weights. The weights are used to mix values. That is the core of attention.

Why the KV Cache Exists

During training, Transformers can process many tokens in parallel. During generation, an autoregressive model emits one token at a time. At step 500, it must attend to the previous 499 tokens. At step 501, it must attend to the previous 500 tokens.

Recomputing keys and values for every previous token on every step would waste enormous compute. Instead, inference systems store keys and values from earlier tokens in a KV cache.

The KV cache is a speed optimization that becomes a memory problem. It prevents repeated computation, but it grows with batch size, context length, layer count, head dimension, and the number of key/value heads.
KV cache size scales with: batch_size x sequence_length x layers x kv_heads x head_dim

The important term for GQA and MQA is kv_heads. Standard attention stores one key head and one value head for every query head. GQA and MQA reduce that number.

Multi-Head Attention: Expressive, but Expensive

Standard Multi-Head Attention, or MHA, gives every query head its own key and value head. If the model has 32 attention heads, it usually has 32 query heads, 32 key heads, and 32 value heads.

MHA
Q1K/V1 Q2K/V2 Q3K/V3 Q4K/V4

This is flexible because every head can learn its own representation space. One head may specialize in syntax, another in long-distance references, another in formatting, and another in local phrase structure.

The cost is that all those key/value heads must be stored and read during decoding. For long contexts, that cache can dominate memory use and memory bandwidth.

Multi-Query Attention: Share One KV Head

Multi-Query Attention, or MQA, keeps many query heads but uses one shared key head and one shared value head. The model can still ask many different questions, but those questions all search the same shared key/value representation.

MQA
Q1Q2Q3Q4 shared K/V

If MHA has 32 KV heads and MQA has 1 KV head, the KV cache for keys and values is roughly 32 times smaller. That can improve latency, increase batch size, and make long-context serving much more practical.

MQA is the aggressive compression point. It is excellent for inference efficiency, but it can lose quality because all query heads must share the same K/V view.

Grouped-Query Attention: The Practical Middle Ground

Grouped-Query Attention, or GQA, sits between MHA and MQA. It divides query heads into groups. Each group shares one key head and one value head.

GQA
Q1Q2Q3Q4 K/V1
Q5Q6Q7Q8 K/V2

For example, a model might use 32 query heads and 8 key/value heads. Each KV head serves 4 query heads. The cache becomes one quarter the size of full MHA, while preserving more representational diversity than MQA.

MHA num_attention_heads = num_key_value_heads
MQA num_key_value_heads = 1
GQA 1 < num_key_value_heads < num_attention_heads
Group size query_heads / kv_heads

MHA vs MQA vs GQA

The entire design space is a tradeoff between attention expressivity and inference efficiency. More KV heads give the model more representational freedom. Fewer KV heads reduce cache size and memory traffic.

MHA

Most expressive

Every query head has its own K/V head. Quality baseline is strong, but the KV cache is largest.

GQA

Balanced

Groups of query heads share K/V heads. Much smaller cache, often close to MHA quality.

MQA

Most efficient

All query heads share one K/V head. Fast and memory-light, but more compressed.

Attention Type Query Heads KV Heads KV Cache Typical Use
MHA Many Same as query heads Largest Maximum flexibility
GQA Many Fewer than query heads Medium Modern LLM serving balance
MQA Many One Smallest Maximum decode efficiency

A simple mental model: MHA gives every analyst private notes, MQA makes everyone share one set of notes, and GQA creates teams that each share their own notes.

Implementation Intuition

In a GQA model, queries and keys/values have different head counts:

Q: [batch, sequence, query_heads, head_dim] K: [batch, sequence, kv_heads, head_dim] V: [batch, sequence, kv_heads, head_dim]

Attention kernels need each query head to use the correct KV head. Conceptually, each KV head is repeated across its assigned query heads. Production systems try to avoid physically copying data, but the logical mapping is easy to understand. 

def repeat_kv(x, num_repeats):
    # x: [batch, sequence, kv_heads, head_dim]
    batch, sequence, kv_heads, head_dim = x.shape
    x = x[:, :, :, None, :]
    x = x.expand(batch, sequence, kv_heads, num_repeats, head_dim)
    return x.reshape(batch, sequence, kv_heads * num_repeats, head_dim)

num_query_heads = 32
num_kv_heads = 8
num_repeats = num_query_heads // num_kv_heads

k = repeat_kv(k, num_repeats)
v = repeat_kv(v, num_repeats)

The decoder then behaves as though it has one K/V view per query head, while the actual cache only stores the smaller number of KV heads.

The Short Version

Standard Multi-Head Attention is expressive but stores a large KV cache. Multi-Query Attention shares one key/value head across all query heads, making inference much cheaper but more compressed. Grouped-Query Attention shares key/value heads within groups, giving a practical balance between quality and serving efficiency.

MHA: many Q, many K/V GQA: many Q, fewer K/V MQA: many Q, one K/V

As context windows grow, the KV cache becomes one of the central constraints in LLM inference. GQA and MQA matter because they attack that constraint directly.

Core takeaway

GQA and MQA do not remove attention. They make attention cheaper by sharing keys and values across query heads, shrinking the KV cache used during generation.
