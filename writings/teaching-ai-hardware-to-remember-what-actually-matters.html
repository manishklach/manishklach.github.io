Methods and Systems for Attention-Sink-Aware SRAM Placement of Key-Value State in Transformer Inference

The Problem Nobody Talks About

When you chat with an AI — ChatGPT, Claude, Gemini, any of them — the system is doing two very different things under the hood. First, it reads your message (the prefill phase). Then it generates each word of the response one at a time (the decode phase). That second part, the decode, is where most of the time goes.

During decode, every single word the AI generates requires it to look back at everything it has read so far. At scale — think 70-billion-parameter models, 32,000-word contexts, hundreds of simultaneous users — that "looking back" means fetching enormous amounts of data from high-bandwidth memory (HBM) on every step. This is called the KV cache problem, and it is one of the primary drivers of cost and latency in modern AI inference.

The standard approach? Store all that data uniformly in HBM and read it uniformly on every step. Simple. Predictable. And, as it turns out, deeply suboptimal — because not all tokens are created equal.

"A small number of token positions attract a disproportionate fraction of attention across every layer, every head, and every decode step. They are, in a real sense, the anchors of the model's memory."

The Attention Sink Phenomenon

Researchers studying how transformer attention actually behaves discovered something fascinating: certain tokens in an input sequence act as attention sinks. The beginning-of-sequence token. The end of a system prompt. Structural separators. These positions receive an outsized, persistent share of attention weight — across layers, across heads, across hundreds of decode steps — even when they carry no obvious semantic importance.

2–8 sink tokens in a typical 8K-token prompt
~40% of total decode attention often directed at sinks
100× SRAM vs HBM latency advantage

Prior work exploited this for algorithmic purposes — StreamingLLM used it to build infinite-context models by keeping sink tokens in a sliding window. But no one had asked the obvious hardware question: if we know in advance which tokens will be accessed most, why are we storing them in the slow memory?

The Invention: Attention-Sink-Aware SRAM Placement

That is the core idea of the patent filed today. During the prefill phase, instead of just building the KV cache and moving on, the system monitors which tokens are attracting attention, scores them, and identifies the sinks. Then — before decode even begins — it migrates the key and value vectors for those tokens from HBM into on-chip SRAM.

  ┌──────────────────────────────────────────────────────┐
  │  t0 [BOS]  t1  t2 ... t55  t56 [ANCHOR]  t57 ... tN  │
  └──────────────────────────────────────────────────────┘
                      ↓  PREFILL + SINK DETECTION
  ┌──────────────────────────────────────────────────────┐
  │  Sink Score S(t) = Σ attention weights received       │
  │  t0: S=0.97  ★ SINK   t2-t55: S<0.18  (non-sink)   │
  │  t56: S=0.83 ★ SINK  t57-tN: S<0.31  (non-sink)   │
  └──────────────────────────────────────────────────────┘
                      ↓  KV PROMOTION (DMA)
  ┌───────────────────┐        ┌──────────────────────────┐
  │  SRAM  (fast)       │        │  HBM  (bulk)             │
  │  t0 KV — pinned   │        │  All 8,000 token KV      │
  │  t56 KV — pinned  │        │  (source of truth)       │
  │  ~1–10 ns access  │        │  ~100–200 ns access      │
  └───────────────────┘        └──────────────────────────┘
                      ↓  DECODE (each step)
  SRAM bypass path  ──→  sink KV reads   (fast  ⚡)
  HBM path          ──→  bulk KV reads   (standard)
  merge outputs     ──→  final attention result
FIG. — Simplified view of the attention-sink-aware KV placement pipeline

During decode, a placement controller intercepts KV read requests. When a request is for a sink token, it routes it to the SRAM path — a nanosecond-scale access. Everything else goes through HBM as normal. The partial attention results from the two paths are merged using numerically stable log-sum-exp accumulation. The model output is mathematically identical. The latency is not.

The Math Behind the Sink Score

The cumulative sink score for a token position is computed as:

FORMULA — SINK SCORE
// Cumulative attention score for token at position t // summed across all layers l, heads h, query positions i S(t) = Σ A(l, h, i, t) ∀ l ∈ [1..L], h ∈ [1..H], i > t // Token t is classified as SINK if: S(t) ≥ T_sink // configurable threshold, e.g. 3.0 normalised over prompt length // Dynamic decay variant (updated each decode step n): E(t, n) = α · E(t, n-1) + (1-α) · B(t, n) // α = decay factor (0.85–0.99), B(t,n) = attention mass at step n

Three Key Embodiments

The patent covers three principal implementations, each with different trade-offs:

Embodiment 1A — Threshold Prefill

Compute S(t) after the full prefill pass. Any token exceeding T_sink gets its KV migrated to SRAM before decode begins. Simple, deterministic, zero decode-phase overhead. Best for short to medium contexts with predictable sink positions.

Embodiment 1B — EMA Dynamic

Use exponential moving average decay to continuously update sink scores during decode. Tokens can be promoted or demoted dynamically. Handles long contexts where sink tokens emerge mid-prompt — common in retrieval-augmented generation and multi-turn instruction following.

Embodiment 1C — Dedicated Hardware Controller

A purpose-built silicon component with a sink score engine, tag array, DMA migration unit, bypass multiplexer, and validity timer. Designed for integration into inference ASICs and AI accelerator chips. Exposes software-configurable registers for all threshold and policy parameters.

Why This Is Harder Than It Sounds

The obvious objection: "SRAM is tiny. You can't fit much in it." Exactly right — and that is precisely why this invention is useful. For a 70-billion-parameter GQA model running at 128K context, the full KV cache is roughly 80 GB. The on-chip SRAM budget might be 96 MB. You clearly cannot cache everything.

But you don't need to. If 2–8 sink tokens account for 30–50% of decode attention traffic, and those tokens can fit in SRAM — a 70B GQA model's KV state for 8 sink tokens across all 80 layers is approximately 2.6 MB — then you capture the majority of the high-frequency access benefit at a tiny fraction of the SRAM budget.

"2.6 MB of well-chosen data in SRAM can have more impact on decode latency than 80 GB of data sitting uniformly in HBM."

The second challenge is correctness. If an SRAM copy becomes stale, or if a token is evicted while still in use, the attention result must remain accurate. The invention handles this through generation counters, validity bits, and an automatic fallback path: on any tag miss, the system transparently falls through to HBM. The model output is always correct.

Beyond the Basic Case

The patent covers a number of important extensions that make it practically useful in real serving systems:

Head-Wise and Layer-Wise Selectivity

Not every attention head is a sink head. Some heads do local pattern matching; others do global retrieval. The invention allows the controller to promote only the key-value slices corresponding to heads that empirically exhibit sink behaviour — reducing SRAM usage by up to 70% versus full-token promotion while retaining most of the latency benefit.

Multi-Tenant Shared Prefix

In serving systems where many concurrent requests share an identical system prompt, the sink tokens for that prompt are promoted once into a shared SRAM region with reference counting. Each new request using the same system prompt gets those sink KV entries for free. Copy-on-write semantics handle per-session mutations. This turns SRAM into a high-speed system-prompt cache — automatically, based on attention behaviour.

Speculative Decoding Integration

In speculative decoding pipelines, the draft model's attention statistics can inform sink classification before the target model's verification pass. Sink KV state staged in SRAM during draft execution is reused during target verification, eliminating a round-trip to HBM.
