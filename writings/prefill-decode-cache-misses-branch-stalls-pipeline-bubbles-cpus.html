Why LLM Prefill and Decode Suffer from Cache Misses, Branch Stalls, and Pipeline Bubbles on CPUs

Published May 10, 2026 · 14 min read

Raw FLOPs and clock speed tell only half the story. The real performance battle in CPU-based LLM inference is about how often the processor is forced to wait while moving weights, activations, and KV cache through a deep transformer stack.

May 2026 · 14 min read · CPU inference / prefill / decode / SIMD

When people first analyze LLM inference, they usually focus on parameter count, tokens per second, memory bandwidth, or maybe even the total number of CPU cores. Those all matter. But once you get close to actual execution, the deeper question becomes much more mechanical: how often is the processor forced to wait?

That waiting shows up in three recurring forms:

  • Cache misses, where the data needed by the core is not present in the cache level it hoped for and must be fetched from farther away.
  • Branch stalls, where the CPU predicts the wrong control-flow path and must recover.
  • Pipeline bubbles, where parts of the execution machinery sit idle because the next instruction or data is not ready.

To reduce the impact of cache misses, branch stalls, and pipeline bubbles in LLM prefill and decode, make the hot path regular, vectorized, cache-local, and prefetchable.

That sentence sounds simple. The reality under it is not. Prefill and decode are not just two names for one workload. They stress the CPU in different ways, and the same hardware pathologies show up differently in each phase.

Prefill and Decode Are Different Workloads

Before talking about microarchitecture, it helps to separate the two major phases of transformer inference.

Prefill

  • Reads the full prompt and builds initial KV cache.
  • Processes many prompt tokens together inside each transformer layer.
  • Has larger matrix operations and more token-level parallelism.
  • Usually gives SIMD units and caches a better chance to stay busy.

Decode

  • Generates one new token at a time.
  • Rereads weights and reuses every layer’s KV cache on each step.
  • Has less fresh-token parallelism and more exposed memory latency.
  • Makes tail latency more visible to the user.

Prefill can often look surprisingly healthy on CPUs because it exposes larger, more structured compute kernels. Decode is usually where the machine starts to show its pain more directly.

Cache Misses

CPUs rely on a hierarchy of caches: L1, L2, L3, and then DRAM beyond that. A cache miss happens when the data the core wants is not present in the expected cache level and must be fetched from farther away.

Mental model
L1 miss -> maybe L2
L2 miss -> maybe L3
L3 miss -> DRAM

The farther out you go, the more expensive the wait.

In LLM inference, the core constantly touches model weights, activations, KV cache, and temporary buffers. If those accesses miss higher-level caches frequently, the processor spends a growing fraction of time waiting on data movement instead of doing useful work.

Cache Misses in Prefill

Prefill is easier to optimize because multiple prompt tokens are processed together. That enables blocked matrix multiplies, more reuse of activation blocks, and better temporal locality for weight tiles. Kernels can often keep a more useful working set warm in cache.

Cache Misses in Decode

Decode is more fragile because the model still rereads many weights while also touching the KV cache of every layer. As context grows, KV grows too. The result is that DRAM latency becomes much more visible in the token loop, especially when locality breaks down.

How to Reduce Their Impact

  • Use cache-friendly tensor layouts.
  • Block and tile matrix operations aggressively.
  • Quantize weights so more fit into upper cache levels.
  • Preserve NUMA locality.
  • Stage hot weight tiles before they are needed.
  • Avoid mixing unrelated hot datasets in the same hottest memory path.
Key Point

Cache misses are reduced mostly by runtime layout, kernel design, and hardware caches. The compiler helps, but it is not the only actor here.

Branch Stalls

Modern CPUs predict branches before they are resolved so the pipeline stays busy. When the prediction is wrong, the CPU discards speculative work and restarts on the correct path. That recovery creates a branch stall.

In hot inference kernels, this matters because the CPU should not repeatedly stop to answer unpredictable questions inside the numerical inner loop.

Branch Stalls in Prefill

Prefill usually has relatively regular control flow: larger matrix kernels, stable iteration spaces, and fewer per-token special cases. That makes it easier to keep branch mispredictions low if the kernels are written properly.

Branch Stalls in Decode

Decode is more vulnerable because sequence-length handling, KV management, runtime scheduling decisions, and dynamic batching logic may leak inward. If those decisions show up in the innermost numerical path, branch stalls start to bite.

How to Reduce Their Impact

  • Move policy logic outside hot kernels.
  • Use specialized kernels for common shapes and cases.
  • Keep inner loops branch-light and shape-stable.
  • Prefer table-driven or precomputed decisions over unpredictable branching.
Key Point

Branch stalls are reduced mostly by kernel design, compiler shaping, and the CPU’s branch predictor.

Pipeline Bubbles

A pipeline bubble is an underutilized cycle where some part of the execution pipeline goes idle because the next useful instruction or data is not ready. Bubbles can come from memory waits, branch recovery, tight dependency chains, or poor scheduling.

The CPU is fast when it has a smooth stream of independent, ready work. It is slow when execution units sit around waiting for the next thing to become available.

Pipeline Bubbles in Prefill

Prefill often gives the processor more room to hide bubbles because the operations are larger, richer in instruction-level parallelism, and better suited to loop unrolling and vector scheduling.

Pipeline Bubbles in Decode

Decode is more exposed. There is less fresh-token parallelism, more visible memory latency, and more sequential dependence. One missing tile or one dependency-heavy path can leave execution units idle for longer than people expect.

How to Reduce Their Impact

  • Vectorize aggressively so the SIMD lanes stay fed.
  • Unroll hot loops carefully.
  • Overlap loads with compute where possible.
  • Reduce dependency chains in the kernel path.
  • Keep more independent work in flight for out-of-order execution to exploit.
Key Point

Pipeline bubbles are reduced mostly by compiler scheduling, kernel design, and hardware scheduling / out-of-order execution.

What “Regular, Vectorized, Cache-Local, and Prefetchable” Really Means

Regular

The hot path should have predictable loops, stable kernel shapes, and few special cases in the inner loop. Regularity helps branch prediction, scheduling, and runtime planning.

Vectorized

The CPU should use SIMD effectively. On x86 this usually means AVX2 or AVX-512. On ARM it usually means NEON, SVE, or SVE2. Without vectorization, CPU inference falls apart quickly.

Cache-Local

The active working set should remain as close to the core as possible. That means reusing weight tiles and activations before eviction, pinning threads sensibly, and avoiding wasteful trips across cache and NUMA boundaries.

Prefetchable

The system should know what data will be needed soon enough to fetch it early. Static model weights are especially suitable for this because their use order is often far more predictable than dynamic KV traffic.

Who Helps Avoid What?

No single layer owns this problem. CPU inference performance is co-produced by algorithm design, runtime behavior, kernel implementation, compiler output, and the hardware itself.

ProblemMain contributors to reducing it
Cache missesMostly runtime + kernel design + hardware caches
Branch stallsKernel design + compiler + branch predictor
Pipeline bubblesCompiler + kernel design + hardware scheduling

This is why “the compiler will handle it” is not a complete answer. The compiler contributes, but the runtime and the kernels often determine whether the compiler even gets a chance to generate good code.

Why Agentic AI Makes This More Important

Agentic systems amplify all of these issues because they often involve long prompts, tool outputs, retrieved documents, scratchpads, and multiple model invocations per user interaction.

  • Prefill cost grows with context size.
  • Decode latency stays visible to users.
  • KV cache keeps growing across turns.
  • Orchestration overhead competes with model latency.

On CPUs, this makes locality, vectorization, and prefetchability even more important. It is not enough to “have enough cores.” The entire hot path has to be shaped so the machine spends more time computing and less time stalled on memory and control flow.

Practical Optimization Checklist

  • Use quantized weights where acceptable.
  • Use cache-blocked GEMM and attention kernels.
  • Keep hot loops branch-light.
  • Pin threads to physical cores and benchmark SMT/HT on and off.
  • Keep NUMA locality correct.
  • Separate weight traffic from KV traffic where possible.
  • Prefetch predictable weight tiles.
  • Move policy logic outside the hottest numerical loops.
  • Minimize decode-path randomness.

Final Takeaway

The biggest misconception in CPU inference is that performance is mostly about core count or clock speed. For LLM prefill and decode, the deeper truth is that performance comes from how well the system turns repetitive tensor workloads into a regular, locality-friendly, vectorized stream that the processor can execute without constantly stalling.

Core count and frequency matter, but the machine wins only when the hot path stops fighting the microarchitecture.
